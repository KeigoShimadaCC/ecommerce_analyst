# Codex eCommerce Analyst — Agent Operating Manual

Read this every session, even if you think you remember it. Codex is a faithful executor of written rules — your leverage is what is encoded here, not what is tribal.

## Project

Codex eCommerce Analyst is a merchant analytics app: a store owner asks a question in plain English; the app uses the Codex SDK to write and run real code against the merchant's data in a per-request snapshot; the verified result and the code are rendered side by side.

See `NORTH_STAR.md` for product vision, the wow bar, the result contract, and milestones.

## Planning Methodology

Phase plans and atomic-task decomposition follow `PLANS.md`. When generating
or revising a phase plan in `phase-plans/`, read `PLANS.md` first and follow
its structure. PLANS.md is the canonical authority for the
orchestrator-subagent dispatch pattern; AGENTS.md remains the authority for
operational rules (commands, sandbox, conventions).

## Prompt Fidelity

Every session must treat its launch prompt as an active contract, not just startup context.

- Before starting work, restate the launch prompt's role, scope, owned files, forbidden files, and definition of done in your private plan.
- During long work, re-check the launch prompt before making scope, architecture, or file-ownership decisions.
- At handoff, report any deviation from the launch prompt explicitly.
- If the launch prompt conflicts with `NORTH_STAR.md`, `AGENTS.md`, `PLANS.md`, or the phase plan, stop and surface the conflict to the parent tier.
- Do not rely on memory of a parent prompt you did not receive; follow only your own launch prompt plus tracked docs.

## Prompt library

The run is driven by three reusable kickoff prompts in `prompts/`, one per tier. They form a chain — each tier is launched from the prompt above it and launches the tier below:

| Prompt | Used by | When | How |
| --- | --- | --- | --- |
| Lead kickoff (local-only) | the human supervisor (Tier 0) | once, at the very start | the supervisor boots the **Codex** Project Lead (Tier 1) with a concrete kickoff prompt — the prompt is local-only and does not ship in the repo; the Lead runs unsandboxed (`danger-full-access`) so it can write git |
| `prompts/phase-orchestrator.md` | the Project Lead (Tier 1) | once per phase | the Lead fills its `{{placeholders}}`, saves to `docs/orchestration/prompts/phase-NN-<name>.md`, and launches it via `scripts/codex-run.sh` |
| `prompts/subtask-worker.md` | the phase orchestrator (Tier 2) | once per atomic task (only when a phase is fanned out) | the orchestrator fills and dispatches one in-session sub-agent per task |

See `PLANS.md` for the tier model and `RUNBOOK.md` for the end-to-end choreography. The unsandboxed, "orchestrator-owned" steps in this manual are performed by the human supervisor (Tier 0) and/or the Codex Project Lead (Tier 1) — the sandboxed tiers (2 and 3) cannot perform them.

**The harness is generated, not shipped.** `scripts/codex-run.sh` and `scripts/codex-report.sh` are generated during pre-build and smoke-tested before P0 (see NORTH_STAR pre-build). A fresh repo is seeded with ONLY the steering docs (`NORTH_STAR.md`, `AGENTS.md`, `PLANS.md`, `prompts/`); everything else (harness, `phase-plans/`, scaffold, env files) is generated by Codex.

## Environment facts (front-load these — DO NOT REDISCOVER)

These are universal across the sandboxed Codex environment:

- **Port 3000 is commonly occupied.** Read `${PORT:-3000}` everywhere a server starts; document `PORT=3001` as the override in the README.
- **The Codex sandbox cannot launch a browser.** Visual smoke is orchestrator-owned (unsandboxed), or runs via Playwright MCP if available. Workers verify via programmatic tests and HTTP route checks only.
- **The Codex sandbox cannot write to `.git` AT ALL** — not commits, not tags, not plumbing (`git write-tree`, `git mktree`, direct object writes). `.git/index.lock` cannot be created; every history operation fails with `Operation not permitted`. This is far broader than "cannot create tags." ALL git history operations (commit, tag, push) are orchestrator-owned. Workers write files, leave the working tree clean-of-intent, stage NOTHING, and record their intended atomic-commit breakdown in the phase report.
- **The Codex sandbox blocks `rm -rf` and chained shell commands.** Use Node `fs` removals; split chained commands into separate steps.
- **Harness launches (`scripts/codex-run.sh`) are Lead-launched only.** A sandboxed session cannot launch a nested harness session: the spawn fails with `Operation not permitted` before model execution. Do not attempt it and do not debug it.
- **The in-session sub-agent facility is the primary Tier-3 dispatch path.** Phase orchestrators dispatch subtask workers as in-session sub-agents, each with a written task brief, and record the per-worker breakdown in the phase report exactly as if they were separate harness sessions.
- **The ledger (`logs/sessions/ledger.tsv`) records one row per harness session.** Worker-level accounting lives in the phase report, not the ledger.
- **Running `next dev` or `next build` generates `web/.next/types/*`** and rewrites `next-env.d.ts`, both of which pollute typecheck. ALWAYS clean `web/.next` (Node `fs.rm` recursive) BEFORE the typecheck gate.
- **`web/.env` is gitignored and does not propagate into git worktrees.** Copy it explicitly into each worktree before running anything.
- **UI verification hierarchy (use the highest rung available, never debug a lower rung):**
  1. Production server evidence from the sandbox: `npm run build`, then `next start`, then authenticated HTTP requests (login POST → cookie → guarded page fetch) proving routes, auth, and rendered markup.
  2. Playwright MCP, only if the browser profile is free; a profile-lock error means skip, not retry.
  3. Lead-level (unsandboxed) or human visual smoke at the phase gate.

  `next dev` is not a verification target: dev-watcher failures (EMFILE, transient 404s) are environment noise whenever the production build serves the same route. Never spend cycles debugging the dev server.

## Sandbox policy

- **Build-time and orchestration Codex sessions** run with `workspace-write`, network access enabled, and approval policy `never`. This is the harness chokepoint (`scripts/codex-run.sh`) configuration.
- **Runtime analysis Codex calls** (the app's engine) run with `workspace-write` confined to a per-request `mkdtemp` directory, network access DISABLED, and approval policy `never`, with a hard timeout of 120 seconds. This policy lives in app code, NOT in `scripts/codex-run.sh`.
- The two policies never share a definition. Conflating them is a security mistake — the runtime engine must never inherit the harness's network access.

## Auth regime (the cost-defining choice)

- **Default: ambient CLI auth.** The Codex SDK drives the already-logged-in local `codex` CLI over stdio. Zero spend on the demo path; no model selection parameter; no reasoning-effort knob.
- **Opt-in API auth:** set `CODEX_AUTH_MODE=api` plus `OPENAI_API_KEY`. The engine fails over to API auth when the configured mode is ambient AND no explicit mode is set AND an API key is available AND the thrown error matches `unauthorized|401|expired|not logged in`.
- **Set the OpenAI dashboard usage cap to $20** before any session, even on ambient auth. The cap is insurance in case the failover puts you on API.
- **If the API path is ever used,** the model resolver MUST validate against the GPT-5 family allowlist (regex `^gpt-5($|-)`) and reject anything else, including `gpt-4o`, `gpt-4o-mini`, `o4-mini`. An unsupported model silently falls back to the expensive default.
- **Nano models** (`gpt-5-nano`, `gpt-5.4-nano`) lack tool support and 400 on tool use. Never select them.
- **Mini models** (`gpt-5-mini`, `gpt-5.4-mini`) satisfied one constraint and silently dropped competing structured-output constraints in testing. For multi-constraint workloads, default to `gpt-5.5`.

## Reasoning effort (only relevant on the API path)

- The SDK default is `medium`, which silently generates 2,000-5,000 reasoning tokens per turn. Never inherit the default; set explicitly.
- On `gpt-5.5`, the floor is `low`. `minimal` is rejected at the model level (the initial misleading error mentioned web_search; the real reason is the model itself does not support `minimal`).
- Set `OPENAI_REASONING_EFFORT=low` in `.env.example` from day one. The adapter must read this variable, validate it against the allowed set, and pass it to the SDK config.
- Coding workloads (analytics, code generation that needs verification) benefit from medium or higher. Dialogue and simple structured emission do not.

## Commit policy

- Atomic Conventional Commits. One coherent change per commit: `feat`, `fix`, `test`, `docs`, `chore`, or `refactor`.
- Do not name any agent tooling vendor in commit messages, files, comments, or filenames.
- **Pinned scan commands.** Assemble the pattern from fragments so this file never contains the literal itself, prove the pattern matches a planted known literal before trusting a CLEAN result, then run both scans:

  ```bash
  PAT="$(printf '%s|%s' 'cla.?''ude' 'anthro.?''pic')"
  # 0) prove the pattern works (must print MATCH):
  printf 'cla%sude' '' | grep -qiE "$PAT" && echo MATCH
  # 1) tracked files (lockfiles excluded):
  git grep -I -i -n -E "$PAT" -- ':!**/package-lock.json' ':!**/*.lock' ':!docs/orchestration/**' && echo DIRTY || echo CLEAN
  # 2) full commit history — messages, authors, emails:
  git log --all --format='%H %an %ae %s %b' | grep -i -E "$PAT" && echo DIRTY || echo CLEAN
  ```

  A CLEAN verdict without the step-0 MATCH line is meaningless and does not pass the gate.
  - The lockfile globs exclude a benign false positive (20× in `package-lock.json` from browserslist/caniuse: a common mobile-OS name whose last five letters collide with one banned substring). The `docs/orchestration/**` glob excludes the deviation log, which legitimately documents the rule. Scope the scan to authored source, not generated lockfiles or orchestration records.
  - **Do not spell the banned words out even with single spaces or punctuation between letters** — the `.?` in the pattern matches a one-character gap, so a spaced-out spelling still trips the scan (a known footgun in orchestrator briefs). In any doc that must reference the rule, name "the two banned vendor substrings" indirectly rather than writing them.
- The default git commit `Co-Authored-By:` trailer may embed banned substrings. Strip it from every commit message before commit. Use `-c commit.gpgsign=false` if needed and verify with `git log -1 --format=%B`.
- Enforce the zero-vendor-literal rule across tracked files, docs, config, comments, filenames, and commit messages.

## Documentation policy

- Architecture decisions live in `docs/decisions/` as ADRs, one per file, immutable after acceptance.
- Phase reports live in `docs/phases/`, one per completed phase, with definition-of-done evidence.
- Add per-module READMEs when a module gains a non-obvious contract or workflow (`web/src/lib/codex/README.md` is required, since the engine is the assignment-critical module).
- Update documentation in the same phase as the related implementation. Documentation written later is documentation written never.
- Every doc has exactly one owner and exactly one update trigger. The orchestrator owns NORTH_STAR.md and AGENTS.md; workers own per-phase reports and module READMEs.
- **No separate progress/status document, and no scratch files at the repo root.** Status lives in the phase report (`docs/phases/<n>.md`) and the deviation log (`docs/orchestration/deviations.md`); there is no `PROGRESS.md`. A per-brief instruction does not stick across sessions. The rule only holds because it lives here.

## Session launch policy

- All build-time and orchestration Codex sessions launch through `scripts/codex-run.sh <label> "<prompt>"`.
- Session logs and JSONL event streams write to `logs/sessions/<label>-<timestamp>.jsonl`.
- The session ledger appends one row per session to `logs/sessions/ledger.tsv` with: timestamp, label, role, model, effort, exit code, wall-clock seconds, input tokens, output tokens, total tokens. Token extraction MUST sum `input_tokens` and `output_tokens` from the JSONL — do NOT grep for `total_tokens`, which does not exist in the event stream.
- Use `scripts/codex-report.sh <phase-prefix>` to summarize phase-level time and token totals.
- **Stall policy.** A session emitting no new events (no commands, messages, or file changes) for ~5 minutes — with the `codex exec` child near-zero CPU — is a transient model-call hang, not progress. The ONLY permitted in-flight intervention is to kill the stalled session and relaunch the IDENTICAL brief once before changing anything. Never hand-edit the worker's partial output; revert any stray edits the killed session made (see File ownership) before relaunch.
- **Do not run two `codex` sessions on the same ambient auth concurrently.** Shared-auth contention is a suspected stall trigger; serialize your build sessions. A concurrent `codex` session from another project on the same machine is not yours to kill — serialize your own and leave others untouched.
- **Out-of-loop sessions** (any session not launched by the Lead loop) run on API auth with the dashboard cap, or only while no Lead/orchestrator session is active on the shared ambient login. They never write tracked files while a phase is active and never touch git; their outputs are handed to the Lead as briefs.
- **Live runtime-SDK turns are Lead-owned.** A sandboxed session cannot execute a real runtime SDK turn (nested execution is blocked before model start). Phase orchestrators verify the engine with the injectable mock runner, the fallback rendering path, and the authenticated HTTP form flow — then stop. The real live SDK turn through the normal app path happens at the Lead review gate, every time, for P2 and later.

## Parallelism policy

- Freeze the shared contract (result schema, DB schema, route boundaries, file ownership) before fanning out.
- Use git worktrees only for file-disjoint subtasks of at least 20 minutes each.
- Worktree paths: `../wt/<phase>-<subtask>`; branch names: `work/<phase>-<subtask>`.
- **MANDATORY worktree pre-flight:** before starting work in any new worktree, run `git fetch origin && git rebase origin/main && git log --oneline -10`. Verify previously-completed phases' merge commits are present. Stale worktrees do not auto-sync and produce silent failures (a worker can write tests against missing production code because their base does not have the merge).
- Copy `web/.env` into each new worktree before running anything (it is gitignored and does not propagate).
- Honestly assess parallelism before paying its tax: if you have fewer than two genuinely file-disjoint workstreams worth more than 20 minutes each, stay serial. For a 4-hour analytics build, serial is often faster end-to-end.
- Integrate after each worktree completes and run `npm run build` (not just typecheck — see Integration below).

## Worker subtypes — delegate aggressively

The orchestrator's context window is the scarce resource, not Codex compute. Default to delegating any bounded, evidence-producing task to an in-session sub-agent and reviewing the distilled result, rather than doing it inline. Keep for yourself only: product/architecture judgment, brief authoring, review-gate decisions, git history (sandbox-blocked for workers — see Environment facts), and user interaction.

- **Verification / QA worker.** *Invoke at every review gate.* Brief: the exact gate commands (typecheck, lint, test, build), the live-turn script, and the DoD checklist for the phase. Returns: pass/fail per gate with command output, plus any defect repro. Evidence lands in the phase report / a gate log. Orchestrator reviews the verdict and decides tag-or-fix — it does not re-run everything inline.
- **Docs / reporting worker.** *Invoke once a phase's evidence exists.* Brief: the evidence to summarize (ledger rows, gate logs, deviations) and the target artifact (phase report, ADR, RUN_REPORT). Returns: the drafted doc citing concrete evidence. Lands in `docs/`. Orchestrator reviews for accuracy and ownership boundaries.
- **Code-reading subagent.** *Invoke to trace a flow or enumerate edge cases without editing.* Brief: the question (e.g. "trace every read path for tenant isolation" / "enumerate engine failure modes") and the files in scope. Returns: a distilled findings list with file:line citations — no edits. Lands inline in the orchestrator's notes. Keeps deep code reading out of the orchestrator's context.
- **Commit / packaging helper (unsandboxed only).** Because the Codex sandbox cannot write to `.git` at all, commits are orchestrator-owned. The mechanical work of grouping a worker's recorded commit breakdown into atomic, vendor-clean Conventional Commits may be delegated to a dedicated UNSANDBOXED helper. Brief: the recorded commit breakdown + the pinned scan commands. Returns: the commit plan executed and the post-push scan result. Orchestrator owns the final push.

## File ownership

- Every phase plan in `phase-plans/<n>.md` declares the files that phase owns and the files that phase is forbidden from touching.
- Workers must check `git diff` against their owned-files list before committing.
- If a worker discovers an architectural need that genuinely requires a file outside the owned list (for example, a server action file because better-sqlite3 cannot be imported into a client component): **STOP.** Surface the diagnosis to the orchestrator. Do not write the forbidden file silently.
- After the orchestrator approves an expansion, amend the phase plan to add the new file to the owned list with a one-line note explaining the architectural need. Then continue.
- **Workers must NEVER edit orchestrator-owned docs (`NORTH_STAR.md`, `AGENTS.md`), even for a benign consistency fix** (e.g. correcting a product name in a heading). A stalled worker may silently rename a heading in `NORTH_STAR.md`; always revert any such edit to the committed state before relaunch. These docs change only by the orchestrator.

## Integration policy

- Run `npm run build` after EVERY merge — not just typecheck, lint, and test. Literal git conflict markers (`<<<<<<< HEAD`) can remain in JSX and still pass typecheck, lint, and test. Only `npm run build` catches them.
- After integration, the Lead runs one real live turn (not just mocks). Mocks do not throw what live APIs throw. Anonymous catches and silent failures only surface in live runs.
- Tag each completed phase as `phase-<n>-complete` (orchestrator-owned — workers cannot write to `.git` at all; see Environment facts).

## Engineering conventions (the universal rules)

These are the conventions every worker follows in every session. They are not optional.

- **No anonymous catch blocks.** Every `try { ... } catch { ... }` must capture the error variable, `console.error` it with context, and either return a structured failure object OR rethrow with a meaningful message. Include at least one test asserting the error path when a dependency throws. Anonymous catches hide live API failures behind generic error messages while mocked tests pass.
- **React falsy renders:** NEVER use `{n && <Component/>}` — it renders a literal `0` when `n` is `0`. Always use `n > 0 ? <Component/> : null`. Add a render test for empty arrays on every list-rendering component.
- **Environment files:** never use `echo >>`. The previous line may lack a trailing newline, gluing values together (a real failure mode: an OPENAI_API_KEY concatenated with the next variable). Use `printf '\nKEY=value\n' >> file` or write the full file with a heredoc (`cat > file <<'EOF' ... EOF`). After every modification verify with `cat -e <file>` (macOS) — every variable line must end with `$`.
- **Dev servers cache environment variables at startup.** After changing `.env`, kill the dev server (`pkill -f "next dev"`) and restart. Save-and-hope does not work.
- **Centralize schema construction.** The Zod result schema lives in ONE module (`web/src/lib/analysis/result.ts`). Every consumer imports the constructed schema; nobody assembles JSON Schema inline at the call site. Add unit tests asserting the schema's shape (root is object, additionalProperties false at every level, every property listed in required) so regressions are caught without live calls.
- **Non-negotiable: file-based results validated by own-Zod, NOT SDK structured outputs.** The runtime Codex thread writes `result.json`; the app reads and validates it with `.strict()`. This is not a preference — it is the contract. Reasons: an own validation boundary the app controls, unit-testable without a live model, and the file doubles as the on-camera proof-of-work. SDK structured outputs are explicitly out (see the next bullet for why, if ever forced).
- **If structured output is ever used** (it should not be for this build): the schema must be a root object with `additionalProperties: false` at EVERY level, every property listed in `required` including optionals, and discriminated unions in place of raw `oneOf`. The SDK rejects schemas that don't meet these constraints.
- **Read-only analytics core.** The runtime engine reads from a snapshot and writes nothing back to the database. If a future feature mutates state, every mutation must have exactly one source of truth (deterministic OR LLM-emitted, never both — mixing them produces phantom duplicates that idempotency-by-ID cannot catch).
- **LLM latency UX.** Show a loading indicator immediately on submit; update the copy at a 15-second threshold ("Codex is still considering..."); disable the input while pending; clear it cleanly when the result lands. This is a convention, not polish.
- **No test may import the production DB client.** Vitest must point at an isolated `file:./test.db` via `vitest.global-setup.ts` (recreated with `prisma db push` per run). Gitignore `test.db`. A test importing the real Prisma client and calling `deleteMany()` on every table will wipe the demo database.
- **No workaround tests.** If a worker writes a test that needs production code that isn't present in their branch, the correct response is to STOP and surface the gap (likely a stale worktree, see Parallelism Policy). It is forbidden to mock the missing production code in the assertion path — that creates green tests that exercise no production logic, manufacturing false confidence.
- **Recovery convention.** Tag each completed phase as `phase-<n>-complete` so recovery is `git reset --hard phase-<n>-complete`.

## Codex SDK runtime — non-negotiable rules

These apply specifically to the runtime engine in `web/src/lib/codex/`. They are stricter than the build-time rules above because the engine is the assignment-critical surface.

- **The app uses `@openai/codex-sdk`.** The app NEVER shells out to the `codex` CLI from runtime code. No `child_process`, no `spawn`, no `codex exec`. Verify with `rg -n 'child_process|spawn\(|codex exec' web/`.
- **Externalize native-binary-backed SDKs in Next.js.** `@openai/codex-sdk` ships a platform binary it resolves relative to its own `__dirname`. By default Next.js bundles server dependencies into `.next/server/chunks`, which breaks that resolution and silently fires the fallback path (`Unable to locate Codex CLI binaries`) instead of running a real turn — even though the dependency and platform binary are installed correctly. Add every native-binary-backed package to `serverExternalPackages` in `next.config.ts` (`serverExternalPackages: ["@openai/codex-sdk", "@openai/codex"]`) so it is required from `node_modules` at runtime, not bundled. Fake-runner unit tests never load the real binary, so this defect surfaces ONLY in a live turn — which is exactly why a real live turn is mandatory DoD.
- **Every runtime SDK call uses a per-request snapshot directory** (`mkdtemp`), populated only from `where: { merchantId }`. The directory is cleaned up in a `finally` block. The merchant ID comes from the authenticated session, never from request input.
- **Every runtime SDK call sets** `sandboxMode: "workspace-write"`, `approvalPolicy: "never"`, network access disabled in both the config and the thread options, `skipGitRepoCheck: true`, and a hard timeout of 120 seconds via `Promise.race` with an `AbortController`.
- **Every runtime SDK call validates `result.json`** against the centralized Zod schema with `.strict()` parsing. Validation failure feeds the error back into a corrective turn (bounded to 2 total attempts).
- **On unrecoverable failure** (timeout, repeated validation failure, dependency throw), persist a schema-valid fallback (`{ answer: "I couldn't complete this analysis reliably.", notes: <reason>, fallback: true }`). Capture whatever generated code and command log exists. The user sees a coherent page, never a stack trace.
- **Make the runner injectable.** The engine takes its runner as an option (`CreateAnalysisEngineOptions.runner`). Tests inject a fake runner that writes canned `result.json` files for fast, deterministic coverage of happy path, retry-then-fallback, timeout, and dependency-throw cases. Real live testing happens at the Lead review gate.
- **Capture the work trail:** walk `turn.items` for `FileChangeItem` and `CommandExecutionItem`, plus a scan of the snapshot directory, so the UI can render the generated code and the command log. Persist these alongside the validated result so saved history shows the work even on the fallback path.
- **The runtime agent must persist its analysis to a script file** (e.g. `analysis.py` / `analysis.mjs`) in the snapshot dir and execute that file — NEVER inline heredocs (`python - <<'PY' ... PY`). Heredocs leave no `FileChangeItem`, so the work-trail walk captures no code file and the on-camera "generated code" panel renders blank, directly defeating NORTH_STAR's "result and code rendered side by side" wow bar. The prompt MUST instruct the agent to write the script before running it; do not rely on synthesizing code from the command log.
- **Every success-path engine test MUST assert `generatedCode` is non-empty.** A blank code panel passes every other gate silently; only this assertion catches the regression at first commit instead of at the live-turn review gate. Pair it with the script-file rule above.

## Lead-owned and supervisor-owned steps (workers must not attempt)

- **Visual browser-based smoke.** The sandbox cannot launch Chromium. Workers verify via programmatic tests, HTTP route checks via `curl`, and Playwright MCP if available; the Lead or human supervisor does the full unsandboxed visual smoke.
- **All git history operations (commit, tag, push).** The sandbox cannot write to `.git` at all (see Environment facts) — not just tags. Workers never run `git commit`, `git tag`, or `git push`, and never stage. They record their intended atomic-commit breakdown in the phase report; the Lead executes vendor-clean commits and tags at the review gate.
- **Cleaning `web/.next` before typecheck.** Standard procedure for any session that ran the app — the worker should call this out as a STEP 0 action and the orchestrator should verify before the typecheck gate.
- **Scanning the full git history for banned vendor literals before push.** Workers do this within their session; the orchestrator does the final scan before the public push.
- **Live runtime-SDK proof.** Phase orchestrators stop at mock-runner coverage, fallback rendering, and authenticated HTTP evidence; the Lead performs the real live runtime-SDK turn through the normal app path.

## Phase boundaries

- Each phase has its own plan in `phase-plans/<n>.md`. Workers stop at the phase boundary defined there. Scope creep into the next phase is forbidden.
- If during execution the phase plan turns out to be incomplete or wrong, surface to the orchestrator and amend the plan before continuing — do not silently expand the scope.
- Definition of done for every phase includes (at minimum): all gates green (typecheck, lint, test, build), a phase report committed to `docs/phases/<n>.md`, and an ADR in `docs/decisions/` for any non-obvious architectural decision made during the phase. **For any phase that touches a runnable surface, DoD additionally requires: `setup` creates `web/.env` with `DATABASE_URL`; authenticated production-server evidence; and — for engine-touching phases — a Lead-owned real live SDK turn whose generated-code panel is non-empty.** Green programmatic gates are necessary but never sufficient; broken login, a silently-faking engine, and a blank code panel have all passed every automated gate and were caught only by running the real app.

## If you are unsure

When the right call is ambiguous, surface to the orchestrator. The orchestrator's review-gate cost is small; a silent wrong turn during a 4-hour build is expensive. The bar: anything that would make a senior engineer reading the resulting git log say "why did you do that?" should be a conversation, not a commit.
